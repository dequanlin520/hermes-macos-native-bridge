## Confirmed CLI Capabilities

`/usr/bin/shortcuts` is available on the research Mac. The installed CLI exposes
help for metadata and execution-oriented commands, including `list`, `run`,
`view` and `sign`. The spike records bounded help output for each supported
probe without recording personal shortcut names.

Observed command surface:

- `shortcuts run <shortcut-name-or-identifier>` with `--input-path`,
  `--output-path` and `--output-type`;
- `shortcuts list` with `--folder-name`, `--folders` and
  `--show-identifiers`;
- `shortcuts view <shortcut-name>`;
- `shortcuts sign --input <input> --output <output>` with `--mode`.

The production Bridge should invoke the CLI by absolute path:

```text
/usr/bin/shortcuts
```

The recommended PATH for the production LaunchAgent is:

```text
/usr/bin:/bin:/usr/sbin:/sbin
```

## LaunchAgent Environment

The active helper runs from `gui/<uid>` and records a sanitized environment:

- UID;
- `HOME=<home>`;
- PATH;
- sanitized TMPDIR;
- executable path;
- sanitized working directory.

The GUI launchd domain was readable, and the temporary LaunchAgent was visible
while loaded.

## LaunchAgent Shortcuts Execution

The active LaunchAgent successfully executed `/usr/bin/shortcuts --help`.
That confirms that a process launched in the current user's GUI launchd domain
can launch the Shortcuts CLI with the bounded environment used by the spike.

The helper PID recorded by the active run was not running after bootout, and
`launchctl print gui/<uid>/<label>` could no longer find the generated service.

## Fixture Execution

The spike checks only for the exact dedicated fixture name:

```text
Hermes Bridge SPK-04 Fixture
```

The fixture was not present during this run, and the installed CLI did not
provide a documented noninteractive fixture creation and import path suitable
for this spike's safety boundary. End-to-end shortcut execution is therefore
blocked until the dedicated fixture is created manually.

## Permission And Consent Behavior

No Shortcuts automation consent prompt was encountered during `shortcuts
--help` from the LaunchAgent. The missing dedicated fixture prevented
validation of consent behavior for an actual shortcut run from the LaunchAgent.

Locked-session behavior was not directly exercised. The evidence supports
current unlocked user-session execution only.

## Timeout And Error Handling

The script wraps Shortcuts invocations in a bounded runner. Timeout handling
sends SIGTERM to the direct child process, waits briefly, then escalates to
SIGKILL for that child only. It records exit code `124`, duration and timeout
state when a timeout is reached.

The active run also invokes one generated missing shortcut name, proving that
nonexistent shortcut execution returns a nonzero result without executing a
personal shortcut. Output is bounded and kept in ignored artifacts.

Production must treat timeout, cancellation, nonzero exit status, stdout and
stderr as structured execution results. It must not log private shortcut input
or output.

## Security Implications

The evidence supports a narrow allowlist model, not arbitrary shortcut
execution. The Bridge should store versioned Shortcut Bindings where each
binding contains one fixed shortcut name or identifier. Callers may request only
an enabled binding and may supply only parameters explicitly declared by that
binding.

Recommended binding fields:

- binding version;
- stable binding ID;
- fixed shortcut name or identifier;
- enabled flag;
- allowed parameter schema;
- timeout;
- output policy;
- audit redaction policy.

Clients must not submit shortcut names, arbitrary executable paths, shell
commands, AppleScript, JXA or GUI automation requests.

## Audit And Redaction Requirements

Safe audit data:

- timestamp;
- binding ID and binding version;
- fixed shortcut name or stable identifier;
- caller identity;
- LaunchAgent/service label;
- executable path;
- timeout setting;
- start and end time;
- exit status;
- timeout/cancel flag;
- expected fixed marker match for fixture tests;
- redacted error classification.

Do not log:

- private shortcut input;
- private shortcut stdout or stderr;
- personal shortcut lists;
- private absolute user paths;
- secrets or credentials.

## Remaining Blockers

- Create the exact dedicated fixture manually and rerun active mode to validate
  end-to-end `shortcuts run` behavior.
- Exercise permission or automation consent behavior with the dedicated fixture.
- Exercise locked-session behavior.
- Exercise a real long-running shortcut timeout case with a dedicated fixture
  designed for that purpose.
